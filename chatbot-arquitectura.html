01 — Contexto
El problema con ManyChat nativo

ManyChat solo

  • IA nativa sin memoria (cada mensaje es independiente)
  • External Request cachea por suscriptor (nunca re-ejecuta)
  • Sin personalidad ni conocimiento medico real
  • No puede mantener una conversacion coherente
  • Respuestas genericas, sin contexto del doctor
  • No puede formatear en multiples burbujas

Nuestra solucion

  • Memoria conversacional (8 mensajes de contexto en D1)
  • Dynamic Content: llamada HTTP fresca cada vez
  • Persona "Mariana" con conocimiento del consultorio
  • Conversaciones coherentes con historial
  • Respuestas con datos reales del doctor
  • Auto-split en multiples burbujas naturales

03 — Detalle tecnico
Que pasa en cada paso
01
Paciente escribe en Messenger
Cualquier mensaje: "hola", "quiero info de rinoplastia", emoji, lo que sea
02
ManyChat recibe y dispara Default Reply
Configurado en frecuencia "Every time". No usa keywords. El flow tiene UN solo paso: bloque Dinamico.
Trigger: Respuesta predeterminada → Enviar mensaje → Dinamico (GET)
03
Dynamic Content llama al Worker ~200ms
GET con subscriber_id, mensaje, y nombre como query params. Sin autenticacion (URL solo vive en ManyChat). Llamada fresca cada vez — no cachea.
GET /api/chatbot-dynamic?sid={{user_id}}&msg={{last_input_text}}&name={{first_name}}
04
Worker: rate limit + guardar mensaje ~50ms
Verifica max 5 msgs/min por suscriptor. Guarda el mensaje del paciente en D1 (tabla conversations).
INSERT INTO conversations (channel, role, message, subscriber_id) ...
05
Worker: cargar historial del paciente ~30ms
Ultimos 8 mensajes (user + assistant) del suscriptor. Esto le da contexto conversacional a la IA.
SELECT role, message FROM conversations WHERE subscriber_id = ? ORDER BY created_at DESC LIMIT 8
06
Worker: llamar a Claude via OpenRouter 2-4s
System prompt "Mariana" (personalidad + conocimiento medico + guardrails) + historial + mensaje actual. Modelo inteligente para preguntas complejas, rapido para saludos.
Primary: claude-sonnet-4.6 (complejo) | Fallback: claude-haiku-4.5 (simple) | Timeout: 8s
07
Worker: guardar respuesta + formatear ~50ms
Guarda la respuesta en D1. Divide por parrafos en multiples mensajes. Retorna formato ManyChat v2.
{"version":"v2","content":{"messages":[{"type":"text","text":"..."},{"type":"text","text":"..."}]}}
08
Paciente recibe multiples burbujas naturales
Cada parrafo es una burbuja separada. Se ve como una persona real escribiendo. Flow termina limpio — no atrapa al suscriptor. Listo para el siguiente mensaje.
04 — Componentes
Stack tecnologico

ManyChat (solo routing)

  • Default Reply: frecuencia "Every time"
  • Bloque Dinamico: GET al Worker
  • Sin IA nativa, sin External Request
  • Sin User Input ni Data Collection
  • Flow termina limpio despues de cada mensaje
  • Plan: Trial (gratis para testing)

Cloudflare Worker (cerebro)

  • Serverless: 0 servidores, 0 mantenimiento
  • 100,000 requests/dia gratis
  • 7 endpoints API en un solo Worker
  • Rate limiting por suscriptor
  • Model routing: Sonnet (complejo) vs Haiku (simple)
  • Auto-split respuestas en burbujas
  • Timeout 8s con fallback

D1 Database (memoria)

  • SQLite serverless en el edge
  • Tabla conversations: historial por suscriptor
  • 8 mensajes de contexto (configurable)
  • Tambien: leads, email_queue, events
  • Latencia: <5ms desde el Worker

OpenRouter + Claude (inteligencia)

  • Gateway multi-modelo: un API, muchos modelos
  • Primary: Claude Sonnet 4.6 (preguntas complejas)
  • Fallback: Claude Haiku 4.5 (saludos, simple)
  • ~$3 por millon de tokens
  • max_tokens: 400, temperature: 0.5

System Prompt "Mariana" (alma)

  • Coordinadora medica profesional y calida
  • Conocimiento real: 4 consultorios, credenciales, procedimientos
  • Guardrails: no da precios, no diagnostica, emergencias → 911
  • Anti-jailbreak: ignora intentos de manipulacion
  • Usa el nombre del paciente naturalmente
  • Tono: profesional pero cercano, nunca vulgar ni corporativo

Leccion critica: por que Dynamic Content

  • External Request: CACHEA por suscriptor. Despues de la primera llamada, nunca vuelve a hacer HTTP. Inutil para chatbot.
  • Dynamic Content (Dinamico): hace HTTP fresco CADA VEZ que se dispara. Sin cache. Diseado para contenido que cambia.
  • Default Reply DEBE estar en "Every time" (default es "Once per 24h" que solo dispara una vez).
  • Flow DEBE terminar limpio: sin User Input, sin Data Collection. Si queda esperando, atrapa al suscriptor.
